# COM-570 — мастер «Register New Object»: демо-сценарии

Пошаговая проходка, чтобы показать всё, что сделано по COM-570 (мастер «Register New Object» для
CRM-датасетов), плюс общие улучшения, которые он подтянул в существующий флоу «Map New Fields».

> UI-надписи и имена полей оставлены на английском — ровно так они видны на экране.

## Чек-лист фич (что показываем)

- Кнопка **Register New Object** на вкладке Objects (только для CRM-датасетов).
- **Шаг 1 — Select Object to Import**: список CRM-объектов. Для HubSpot показываются поддерживаемые
  типы записей (Company / Contact / Deal) — это новая бэкенд-энумерация (раньше HubSpot объекты не
  перечислял).
- **Шаг 2 — Enter Object Details**:
  - **Record ID** — мультиселект с поиском и **чекбоксами**; выбор поля **автоматически включает** его
    строку в таблице с преднастроенным «Create New Field».
  - **Таблица полей** — тоггл на каждое поле, поиск, фильтр по типу, **Map to Field**, встроенный
    редактор поля (тип данных / описание) по иконке-карандашу.
- **Шаг 3 — Map Object Relationships**:
  - Слева — parent (по умолчанию Person-объект), справа — child (по умолчанию новый объект).
  - Стрелка между объектами — **переключатель кардинальности** (One-to-Many ⇄ One-to-One) с тултипом.
  - Селекты объектов **ограничены объектами, уже входящими в дерево merge strategy** (+ сам новый
    объект), чтобы нельзя было создать «висящую» связь. Ширина стрелок — как в дизайне (40px).
  - **Валидация дублей**: нельзя продолжить, если один и тот же объект указан как child дважды.
- **Шаг 4 — Register New Object**: компактная карточка подтверждения (того же размера, что и
  подтверждение «Map New Fields»). Создаёт объект (с валидным ключом классификации), его поля и
  вплетает связи в merge strategy датасета. Устойчив к повторному нажатию (retry-safe).
- **Общее**: контрол Record ID и поверхность маппинга полей — тот же компонент, что в существующей
  модалке «Map New Fields»; поведение «авто-включение по Record ID» добавлено и туда.

## Окружение / предусловия

- Аккаунт **5 — tvv_hs** (HubSpot), датасет **HubSpot CRM** (`tvv-hs-hubspot`).
- URL: `localhost:5173/accounts/5/datasets-compass/tvv-hs-hubspot`
- Бэкенд (:9080) пересобран, чтобы энумерация объектов HubSpot работала; фронтенд dev-сервер на :5173.
- Нужен **живой коннект к HubSpot** (песочница Пата) — шаг 2 подгружает поля объекта из HubSpot
  вживую.
- Текущее дерево merge strategy этого датасета = только **HubSpot Contact** (`hs-contact-crm`). Это
  единственный объект «в дереве», к которому можно цеплять связи.
- Демо-объект ниже — **Deal**: его ключ классификации (`deal`) свободен, поэтому его можно
  регистрировать повторно без коллизий. (Ключи Company/Contact могут уже существовать после прошлых
  прогонов; чтобы переиспользовать их, сначала удалите leftover-классификацию — см. «Сброс» внизу.)

## Сценарий A — регистрация нового объекта от начала до конца (основной флоу)

1. Открой датасет, вкладка **Objects**. Нажми **Register New Object** (справа вверху; появляется только
   для CRM-датасетов — для CSV-датасета вместо неё будет «Create New Object»).
2. **Шаг 1 — Select Object to Import**: в списке показаны **Company, Contact, Deal**. Через поиск
   отфильтруй и кликни **Deal**.
3. **Шаг 2 — Enter Object Details**:
   - В шапке видно `API Name: DEAL`.
   - **Record ID**: кликни по полю → откроется выпадающий список с **чекбоксами** и поиском. Введи
     текст для фильтра, отметь идентификатор сделки (например **Record ID** / `hs_object_id`). Обрати
     внимание: соответствующая строка в таблице ниже **включается автоматически**, с уже выбранным
     **Create New Field**.
   - **Fields (N)**: включи несколько полей для маппинга (пример — бери эти или похожие из списка):

     | Toggle | Label (пример) | API name (пример) | Type | Map to Field |
     |---|---|---|---|---|
     | on | Deal Name | `dealname` | String | Create New Field |
     | on | Amount | `amount` | Number | Create New Field |
     | on | Close Date | `closedate` | Date | Create New Field |
     | on | Deal Stage | `dealstage` | Enumeration | Create New Field |

   - Для одной строки нажми **иконку-карандаш** → откроется встроенный редактор → подтверди/поправь
     **тип поля** и добавь короткое **описание** → **Save Changes**. (Переименование: кликни по
     лейблу, чтобы изменить отображаемое имя.)
   - Проверь **Search fields…** и **фильтр по типу** (например только «Number»), чтобы показать, что
     фильтрация работает.
   - Нажми **Next Step**.
4. **Шаг 3 — Map Object Relationships** (опционально, но показываем): связываем новый Deal с
   существующим Contact, чтобы сделки группировались под своим контактом.
   - Нажми **+ Add New Relationship** → появится строка.
   - **Строка Object**: слева (parent) = **HubSpot Contact** (по умолчанию, в дереве), справа (child)
     = **Deal** (по умолчанию, новый объект). При наведении на стрелку-коннектор — тултип
     **One to Many** (у контакта много сделок); кликни один раз — переключится на **One to One**,
     кликни ещё — вернётся на One to Many.
   - **Строка Field**: parent field = **Id** (ключевое поле Contact), child field = поле Deal, которое
     ты включил на шаге 2 и которое ссылается на контакт (или **Record ID** сделки для демо). Оба
     селекта полей с поиском.
   - Нажми **Next Step**.
   - Подробно про UI связей и защиты — в **Сценарии C**.
5. **Шаг 4 — Register New Object**: компактная карточка подтверждения показывает `Deal will be
   registered as a new object`, `N field(s) mapped`, `1 relationship(s)`. Нажми **Register**. Карточка
   подтверждения просит после регистрации запустить новый CRM sync — данные визард только сохраняет, не
   процессит.
6. **Результат**: тост об успехе; мастер закрывается; новый объект **Deal** появляется на вкладке
   Objects с замапленными полями, а связь сохранена в merge strategy датасета (видно на вкладке
   **Relationships**).

## Сценарий B — UX Record ID (мультиселект с поиском + чекбоксы + авто-включение)

На шаге 2:

1. Открой **Record ID** → убедись, что у каждого пункта есть **чекбокс**, список **с поиском**, и
   поддерживается **множественный** выбор (выпадашка остаётся открытой при отметке).
2. Отметь поле → его **строка в таблице включается автоматически**, с преднастроенным
   **Create New Field** (поведение «при выборе Record ID авто-включаем строки»).
3. Сними отметку в Record ID → строка остаётся как была (снятие не выключает её принудительно).

Этот же контрол и поведение переиспользуются в существующей модалке **Map New Fields** (Сценарий D).

## Сценарий C — связи (relationships) подробно

Как связи устроены здесь (одной строкой): связь — это **parent → child**, соединённые парой полей
(значение parent-поля = значение child-поля). Все связи датасета живут в **одном связном дереве**
(merge strategy). Мастер вплетает новый объект в это дерево.

### C1 — Анатомия строки связи

В каждой строке две линии:
- **Object**: `[parent] → [child]`. Слева всегда parent (по умолчанию Person-объект, здесь
  **HubSpot Contact**); справа child (по умолчанию новый объект, здесь **Deal**).
- **Field**: `[parent field] → [child field]`. Ключ соединения. Пример: `Id → Record ID` означает
  «сделка принадлежит контакту, у которого Id равен Record ID сделки».
- Обе стрелки **однонаправленные (→)**. **Стрелка в строке Object кликабельна** — она переключает
  **кардинальность** и показывает тултип:
  - **One to Many** (по умолчанию): один контакт → много сделок.
  - **One to One**: кликни по стрелке, чтобы переключить; кликни ещё раз, чтобы вернуть.
- Слева стрелки-реордера и иконка корзины — переставляют / удаляют строку.

### C2 — Ограничение по дереву (почему некоторые объекты не в списке)

- Текущее дерево merge strategy у `tvv-hs-hubspot` = только **HubSpot Contact**. Поэтому оба селекта
  объектов предлагают только **HubSpot Contact** (в дереве) и **Deal** (новый объект).
- **HubSpot Company** и **HubSpot Deal** **не** предлагаются — они пока не связаны с деревом. Связь с
  ними расколола бы датасет на два отдельных дерева, а это запрещено. Чтобы их использовать, сначала
  подключи их на вкладке **Relationships**.

### C3 — Валидация дублей child

- Добавь **две** строки с **одним и тем же child** (например обе `… → Deal`) → нажми **Next Step** →
  появится ошибка **«Duplicate relationship»** и продолжить нельзя. Объект может быть child только
  один раз (priority merging не допускается).

### C4 — Крайний случай: пустое дерево

- Если у датасета вообще нет связанных объектов, **+ Add New Relationship** **задизейблена** с
  подсказкой зарегистрировать объект и связать его на вкладке Relationships (цеплять пока не к чему).

### C5 — Финальная страховка (форма на сервере)

- При Register мастер перестраивает дерево merge с новыми связями. Если результат всё равно оставил бы
  какой-то объект «висящим», мастер **сохраняет созданный объект**, но **пропускает связи** и
  показывает *«Relationships not saved … Register the object, then link it from the Relationships
  tab.»* Данные не портятся.

### C6 — Проверка результата

- После успешной регистрации открой вкладку **Relationships** → в графе видно **HubSpot Contact →
  Deal** (parent → child) с выбранной кардинальностью. Это та же merge strategy, которую редактирует
  полный редактор связей.

## Сценарий D — та же поверхность в «Map New Fields» (переиспользование)

1. На вкладке Objects разверни существующий объект и нажми **Map New Fields**.
2. Покажи **ту же** таблицу маппинга полей, контрол **Record ID** (с поиском + чекбоксами) и
   поведение **авто-включения по Record ID** — это доказывает, что мастер переиспользует существующую,
   отревьюенную поверхность.
3. На **Person**-классификации (HubSpot Contact) выпадашка **Map to Field** дополнительно показывает
   **known fields** Windfall (email, name, …). Именно здесь демонстрируется маппинг на known-fields —
   почему у нового объекта мастера их не видно, см. оговорку ниже.

## Оговорки / известные ограничения (упомянуть, если спросят)

- Объект, зарегистрированный мастером, создаётся как **Custom**, а список known-fields в **Map to
  Field** фильтруется по типу объекта. У Custom нет known-fields, поэтому «Map to Field» в мастере
  показывает только **Create New Field**. Подсказки known-fields видны в **Map New Fields** на
  Person/Transaction-объектах (Сценарий D). (Селектор типа прототипировали и откатили — его нет в
  дизайнах.)
- **Шаг 2 требует живого коннекта к HubSpot** для загрузки полей объекта.
- Энумерация HubSpot сейчас перечисляет поддерживаемые типы записей (Company / Contact / Deal), а не
  произвольные кастомные объекты HubSpot.

## Сброс (только если переиспользуешь Company/Contact или для чистого перезапуска)

Мастер не удаляет объекты. Чтобы заново зарегистрировать объект, чей ключ уже существует (например
`company` после прошлого прогона), удали leftover-классификацию напрямую в локальной БД hamster и
повтори:

```sql
-- hamster PG: dev:password@localhost:15433/hamster ; account_id — это match-id строкой
DELETE FROM classification WHERE account_id = '5' AND key = 'company';
```

(Нужно только для чистого демо; флоу с Deal выше сброса не требует.)
