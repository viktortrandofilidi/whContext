# COM-572 — CRM Registry «Map to Field» — тест-сценарии

Стори: Compass Dataset Mgmt | Add Field in CRM Datasources Should Open CRM Registry
Ветка: `viktor/COM-572/crm-registry-map-fields` (portal-admin)
Что тестируем: модалку **«Enter Object Details»**, которая открывается по кнопке **«Map fields»**
на классификации CRM-датасета и позволяет замапить неотмапленные CRM-поля на Windfall-поля
или на новое кастомное поле.

---

## Предусловия / тестовые данные

Нужен аккаунт с **CRM-интегрированным** Compass-датасетом. Проще всего на dev/staging, где есть
живое Salesforce/HubSpot-подключение (локально нужен рабочий CRM-connect, иначе метаданные CRM не
загрузятся).

| Требование | Как проверить, что подходит |
|---|---|
| Датасет CRM-интегрированный | у датасета `dataSourceKey` не null (SALESFORCE / HUBSPOT) |
| Классификация связана с CRM-объектом | у классификации `crmEntityKey` не null (напр. `Account`, `Contact`, `Transaction`) |
| Есть неотмапленные CRM-поля | у CRM-объекта есть поля, которых ещё нет в классификации |
| Есть уже отмапленные поля | чтобы проверить, что они отфильтрованы |
| Доступен каталог Windfall-полей | `/compass/proxy/tags/catalog/by-category` отдаёт теги под рутом этой классификации |

Контрольный «не-CRM» кейс: обычный CSV-датасет (`dataSourceKey = null`) — для регрессии
(должна открываться старая модалка одиночного создания поля, а не CRM-реестр).

---

## Группа A. Точка входа / роутинг

**A1. CRM-датасет → открывается CRM-реестр**
1. Открыть CRM-датасет → развернуть аккордеон классификации.
2. Нажать **«Map fields»**.
- ✅ Открывается модалка с заголовком **«Enter Object Details»**.
- ✅ Вверху видно `API Name: <объект>` и бейдж `N fields`.

**A2. Не-CRM датасет → старый флоу (регрессия)**
1. Открыть CSV-датасет → классификация → «Map fields».
- ✅ Открывается прежняя модалка **«Create New Field»** (одно поле), НЕ CRM-реестр.

---

## Группа B. Загрузка и ошибки

**B1. Состояние загрузки** — при открытии видно лоадер и текст «Loading CRM fields…».

**B2. Ошибка загрузки CRM** — если метаданные CRM не пришли (объект/подключение недоступны):
- ✅ Красный алерт `Failed to load CRM fields for "<объект>". Please try again.`

**B3. Классификация без CRM-объекта** (`crmEntityKey = null`, если так открыли):
- ✅ Алерт `This classification is not linked to a CRM object.`

---

## Группа C. Список полей

**C1. Показаны только неотмапленные поля** — поля, уже существующие в классификации (по
санитизированному имени), в таблице отсутствуют. Нельзя размапить существующее поле из этого флоу.

**C2. Колонки и источник данных** — `[тумблер] | Label | API Name | Column Type`. Label = CRM display
name, API Name = CRM api name, Column Type = тип из CRM (в верхнем регистре).

**C3. Поиск** — поле «Search fields...» фильтрует по Label и API Name.

**C4. Фильтр по типу** — селектор «All Types» фильтрует строки по Column Type.

**C5. Пустое состояние** — если под фильтр ничего не попало → строка «No unmapped fields.».

**C6. Record ID** — селектор «Record ID» вверху даёт выбрать поле-идентификатор из списка
неотмапленных (searchable, clearable).

---

## Группа D. Dropdown «Map to Field» (ядро стори)

**D1. Включение строки открывает редактор**
1. Включить тумблер на строке.
- ✅ В колонке «Map to Field» появляется dropdown со значением **«Create New Field»** (дефолт) и
  иконка-карандаш справа.
- ✅ Сразу раскрывается инлайн-редактор под строкой (режим создания нового поля).

**D2. Содержимое dropdown**
1. Открыть dropdown «Map to Field».
- ✅ Windfall-поля **сгруппированы по категориям** каталога (заголовки групп).
- ✅ В самом низу — пункт **«Create New Field»**.
- ✅ Dropdown searchable — поиск фильтрует по всем группам.

**D3. Выбор Windfall-поля → тип залочен**
1. В dropdown выбрать известное Windfall-поле.
- ✅ Редактор пересобирается: имя = имя поля, **Field Type задизейблен** (с иконкой замка), тип = тип
  известного поля.
- ✅ Description и «Available for» доступны.

**D4. Выбор «Create New Field» → тип редактируемый**
1. В dropdown выбрать «Create New Field».
- ✅ Field Type **редактируемый**, по умолчанию выведен из типа CRM-поля (boolean→Boolean,
  currency/double→Float, int→Integer, date→Date, прочее→String).
- ✅ Имя по умолчанию = Label CRM-поля.

**D5. «Create New Field» недоступен при раскрытом редакторе**
1. При включённой строке с открытым редактором открыть dropdown.
- ✅ Пункт «Create New Field» **серый/некликабельный**.
2. Нажать Cancel или Save Changes (редактор закрылся) → снова открыть dropdown.
- ✅ «Create New Field» снова доступен.

**D6. Карандаш переоткрывает редактор**
1. Сохранить маппинг (редактор закрылся) → нажать карандаш справа от dropdown.
- ✅ Редактор снова открывается с ранее сохранёнными значениями.

**D7. Смена цели сбрасывает несохранённое**
1. Выбрать known-поле A, начать заполнять → в dropdown переключить на known-поле B (или Create New).
- ✅ Редактор пересобирается под новую цель; предыдущий несохранённый результат сброшен (строку надо
  сохранить заново).

---

## Группа E. Мульти-выбор и сохранение

**E1. Несколько полей одновременно** — можно включить несколько строк. В каждый момент открыт
редактор только одной строки.

**E2. Незаполненная строка блокирует «Next Step»**
1. Включить строку, но НЕ сохранить редактор → нажать **Next Step**.
- ✅ Предупреждение «Finish editing / Save changes for every enabled field before continuing.».

**E3. Пустой выбор**
1. Не включать ни одной строки → «Next Step».
- ✅ Предупреждение «No fields selected».

**E4. Успешное создание**
1. Включить 2–3 поля, у каждого выбрать цель и нажать Save Changes → «Next Step».
- ✅ Зелёное уведомление «Fields mapped — N field(s) added to <классификация>».
- ✅ Модалка закрывается.
- ✅ Новые поля появились в классификации (таблица полей обновилась без перезагрузки).

**E5. Record ID → ключевое поле**
1. Выбрать поле в «Record ID», включить его и ещё пару, сохранить → «Next Step».
- ✅ Поле, выбранное как Record ID, создано как key (isKey / mapping required).

**E6. Идемпотентность списка**
1. После E4 снова открыть «Map fields» на той же классификации.
- ✅ Только что созданные поля больше НЕ показываются (они теперь отмаплены).

**E7. Типы маппинга**
- known-поле → создаётся поле с тегом выбранного каталожного поля;
- «Create New Field» → создаётся кастомное поле с выбранным типом (без каталожного тега).

---

## Группа F. Сброс состояния

**F1. Cancel / Back / крестик** — закрыть модалку с частично заполненными строками → открыть заново.
- ✅ Всё сброшено: тумблеры, выбор в dropdown, поиск, фильтр типа, Record ID.

---

## Известные ограничения (НЕ баги — не заводить)

1. **«Available for» (Application / Modelling) НЕ сохраняется.** Галочки собираются формой, но не
   пишутся в бэкенд (нет слота в контракте `:amend`). Сейчас косметика; персист — отдельная работа
   (usage-семантик-теги, COM-579).
2. **Скоуп known-полей.** Dropdown показывает Windfall-поля только под рутом текущей классификации
   (`tagPathRoot`) — сохраняет прежнее поведение. В Figma нарисовано несколько групп разных
   классификаций; это иллюстративно. Расширение скоупа — отдельное решение.
3. **Не реализовано из макета (осознанно):** подтверждение по Enter + автофокус следующего поля;
   тумблер «Permissions: Write» (был бы no-op — ранее ревьюер убрал no-op switch). «Next Step» =
   финальный сабмит (второго шага визарда нет), «Back» = закрыть.
